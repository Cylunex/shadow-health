# 周/月复盘播报

把快照数字讲成有对比、有归因、有下一步的复盘，并把用户的主观感受收进档案。

## 流程

1. `query_weekly_report()` 缺省即上一完整周（月报用 `query_monthly_report()`）。
   `complete=false` 表示该周期还没走完——播报时必须注明「进行中」。
2. 想给环比再取上一期：`query_weekly_report(week="2026-W27")`（ISO 周）。
3. 播报结构（按有数据的说，缺的跳过）：
   - **一句话总评**（进步/持平/走弱，从体重趋势与负荷变化判断）
   - 体重与体成分变化；**能量账本**（gap_sum → 理论 vs 实际对账——两者差得远
     通常是漏记，如实点出「本周有 N 天没记饮食，账对不准」）
   - 训练：次数/分钟/sRPE 负荷/**ACWR**（>1.5 提醒下周减量，<0.8 说可以加）
   - 睡眠：均时长/达标夜/均入睡时刻（波动大就点名作息漂移）
   - 打卡率、日均步数、心情均分
4. 播完问一句开放式复盘（「这周最难坚持的是什么？」），用户的回复**原话**
   通过记录员流程之外的方式保留——目前周/月复盘写入仅 UI，提示用户
   「可以到 报告 → 周报 里把这段感受存进手写复盘」。
5. 用户想要深度分析 → `run_analysis(days=30)` 触发后台报告（1-2 分钟），
   之后 `get_analysis()` 取回 markdown 播报要点（job=running 就说稍等，
   **轮询间隔 ≥15 秒且不要重复触发**）。

## 铁律

- 所有数字来自快照回执；两期对比只对比同口径字段。
- 旧周期快照可能缺新字段（acwr/睡眠/能量账本）——缺就不提，不要说「没有数据」
  显得像坏了。
- 复盘语气：趋势与调整语言，不评判（「日均超了 X」→「实际维持热量比目标高 X，
  下周目标可以顺势调到 Y」）。
- 本技能只读；改目标热量让用户去 报告 → 每周 Check-in 一键应用，或设置页改。

## 示例播报

> 上周（W27）小结：体重 -0.4 kg（71.9 → 71.5），能量账本 -2,800 kcal，理论
> -0.36 kg，和实际对得上，记录质量很好。训练 5 次 385 分钟，负荷 1,240，
> ACWR 1.1 适中。睡眠均 6.9h，3 晚达标，均入睡 23:40——比前一周晚了半小时。
> 打卡率 86%。这周最难坚持的是什么？
