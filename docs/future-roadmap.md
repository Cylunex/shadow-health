# shadow-health 面向未来的机会清单

## 1. 行业在往哪走

2025-26 年健康应用的共同转向，是从「记录与展示」走向「个人基线 → 偏离检测 → 主动建议」的闭环：Apple Vitals/Training Load、三星 Energy Score、Fitbit Daily Readiness、MacroFactor 自适应 TDEE、Keep AI 教练，本质都是同一件事——用你自己的历史数据建基线，把多源被动数据合成单一可行动结论，并在偏离时主动找你，而不是等你来看图。对自托管单用户场景这是罕见利好：这些能力约九成是纯本地统计（滚动基线、EWMA、窗口对比、加权合成、配对相关），不需要云也不需要大模型常驻，天然符合断网优先原则；而本地数据盘点证实 shadow-health「采集面远大于利用面」——手表夜间心率/睡眠分期/SpO2、sRPE、体脂秤五项体成分、逐日能量缺口、mood/energy/晨勃都已在 PG 里，却大多只有一行展示。因此最大机会不是接新数据，而是在已有数据上补一层「基线引擎 + 分数合成 + 趋势巡检 + 主动推送」。方向性教训来自 Apple 砍掉大一统 AI 教练 Project Mulberry：单人项目应走「一个个小而硬的本地洞察」，已有的 AI 双通道只做可选的叙述与解读层；敏感数据（体检报告、体态照片）留在 NAS 不出网，反而是相对云 App 的核心竞争力。

## 3. 明确淘汰清单

| 不做 | 原因 |
|---|---|
| CGM 连续血糖联动 | 专有硬件国内可得性差、厂商云 API 强在线依赖，且用户目标是减脂+体能而非代谢病管理（nutrition 路已判定） |
| 步行稳定性/跌倒风险 | 依赖 iPhone 专有步态算法与原始 IMU，三星直读拿不到；体能良好的成年男性收益趋近于零 |
| 大一统实时 AI 教练 / 生成式实时语音激励（Workout Buddy 式） | 违背断网原则、NAS 算力不划算；Apple 自己砍掉 Project Mulberry 是反向印证——改走「小而硬的本地洞察 + 规则化 TTS」 |
| 84 种微量营养素全量追踪 | 单用户录入负担会反噬记录率，数据维护成本高；只取纤维/钠/糖三个高价值字段（C5） |
| 视频跟练课程 | 版权、存储、制作成本三重不可行；TTS 跟练播放器（B1）以 1/10 成本覆盖核心价值 |
| 抗氧化指数（皮肤传感器） | Galaxy Watch 专有传感器不可得；目标已转译为饮食多样性分数并入 C6/C9 |
| 天气 API 作为核心依赖 | 外网强依赖违反硬原则；只允许做成可缺失的可选项，或用局域网自有传感器/FIT 文件温度字段替代 |
| 常模对比类功能（同龄同性别百分位、睡眠动物常模） | 无人群数据库可用；一律改为「个人基线对比」，这也正是行业方向 |
| 云端体态评估作为唯一路径 | 敏感全身照出网违背自托管定位；仅保留本地关键点路线（H4），AI 点评作可选 |
| 药物相互作用库 | 单用户收益极低，维护成本高；补剂只做清单+打卡（G9） |
| 社交/社区/排行榜/电商/会员/直播课/线上赛报名 | 单用户场景无意义（题设已剔除，此处确认）；其激励价值由 H1 成就体系承接 |

## 4. 推荐 V6 批次（按优先级）
